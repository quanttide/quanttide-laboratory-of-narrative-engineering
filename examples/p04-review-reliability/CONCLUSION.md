# p04 实验结果

## 结论

**Review（意图理解）高度可靠，适合做 3R 的第一环。**

## 关键数据

### 稳定性

| 文章 | 体裁一致性 | 意图波动 | 阶段判断 |
|------|-----------|---------|---------|
| A 咖啡厅重逢 | 5/5 一致 | 波动在"重逢剧情"vs"暧昧氛围"间，无矛盾 | 5/5 判为成稿（实际是初稿——有意义偏差） |
| B 便利店闲坐 | 5/5 一致 | 5/5 完全一致"推进人物关系，展示内心情感" | 5/5 判为成稿 |
| C 第六章 | 5/5 一致 | 波动在"CP互动"vs"粉丝反应"间，无矛盾 | 5/5 判为成稿（实际是初稿——有意义偏差） |

### 区分度

**聚类 100%**（15/15）——15 次 Review 输出完美分成 3 组，每组对应一篇文章。

### 一个有趣的发现

A 和 C 都是初稿，但 Review 全部判为"成稿"。这不是 LLM 判断错误——而是这两篇初稿本身的完成度已经很高。Review 判断的是**文本呈现出来的阶段**，而不是"文件系统上的标签"。这意味着：
- Review 可以作为一个客观的**完成度量尺**
- 如果产品中某篇标注为"初稿"的文章被 Review 判为"成稿"，说明它其实不需要再改

## 对产品的启示

1. **Review 可以作为独立服务**——不需要缓存或状态，每次调用输出稳定
2. **体裁分类粒度可以更粗**——3 篇文章中 A 和 B 都是"重逢场景"（B 其实是便利店闲坐），说明"重逢场景"这个标签太宽，但对理解意图来说不影响
3. **意图的措辞波动可接受**——核心理解一致的前提下，措辞不同不影响 Reflect（Reflect 需要的是理解，不是精确措辞）
4. **阶段判断可以作为"完成度检测器"**——这可能是 Review 在产品中最直接的功能

- 总调用：15 次 LLM API（3 篇文章 × 5 次）
- 聚类调用：1 次 LLM API
