# E4-7 — 基于多画像评价的局部修改建议（Rewrite）

> 对应能力：Rewrite（修复层）

## 动机

p16-reader 能告诉作者"P3 觉得这个场景套路感强"——但作者需要的是具体怎么改。

E4-7 的核心假设是：**给定文本 + 多画像评分分布 + 目标画像，LLM 能定位薄弱点并生成针对该画像的结构性修改，且修改后目标画像的评分会提升。**

## 实验范围

E4-6（Reflect/视角译码）已被砍掉。E4-7 自包含 Reflect+Rewrite 两个步骤：

```
E4-7 Step 1:      找出"哪里不对"（定位薄弱点）
E4-7 Step 2:      生成"怎么改"（修改版本）
E4-7 Step 3:      验证"改好了吗"（E4-1 评价引擎 before/after 对比）
```

## 核心洞察

从 p10（母题-风格交叉）和 p14（片段内部情节建议）的经验看，有效的修改建议需要满足：

1. **可执行**：作者拿到后知道在文本里改哪个位置、改什么
2. **有理由**：建议附带"为什么要这么改"（基于读者画像偏好）
3. **最小改动**：只改薄弱点，不重写全篇
4. **选择性**：不是所有文本都需要改，只有目标画像评分低于阈值的维度才生成修改

## 实验设计

### 输入

| 来源 | 内容 | 格式 |
|------|------|------|
| `e4-1_raw.json` | 5 画像 × 6 文本 × 3 call 的完整评分 | JSON |
| `assets/fiction/4_成稿/` | 6 篇测试文本的全文 | Markdown |
| `profiles.json` | 5 个画像的行为锚定定义 | JSON |

### Step 1: 薄弱点定位（Reflect，自包含）

对每篇文本 × 目标画像：

1. 找到目标画像评分最低的维度（如 P3 的 cliche_level=4.0，低于其他画像的均值）
2. 如果所有维度评分都 ≥ 目标画像在这 6 篇文本上的历史均值（来自 e4-1_summary.json），则判定"无需修改"，跳过该文本

```
P3 历史均值（来自 Phase I）:
  writing_quality: 5.11
  emotional_impact: 5.39
  character_realism: 5.94
  cliche_level: 1.72

新的 2.1（展会再遇）P3 评分：
  cliche_level: 4.0 ← 远高于历史均值 1.72，差距 = 2.28
  → 判定：cliche_level 需修改
```

输出：{text_id, target_profile, target_dimension, current_score, target_score, gap, relevant_segment}

### Step 2: 生成修改（Rewrite）

输入：
1. 文本片段（薄弱点相关的段落）
2. Step 1 输出的薄弱点分析（画像 + 维度 + 当前评分 + 目标评分）
3. 目标画像的行为锚定
4. 约束：最小改动、保持场景结构

输出：

```json
{
  "target_profile": "P3",
  "target_dimension": "cliche_level",
  "current_score": 4.0,
  "target_score": 2.0,
  "original_segment": "原文段落...",
  "suggested_rewrite": "改写后的段落...",
  "what_changed": ["删去了第三句的套路化对白", "加入了林远亭的犹豫"],
  "why_for_reader": "P3 对套路最敏感——原版对话模式是'你先说我拒绝你再说我接受'，这是老书虫一眼识破的结构",
  "minimal_changes": true
}
```

对每篇文本 × 目标画像，生成 2 个不同 prompt 风格的版本。

### Step 3: 验证（用 E4-1 评价引擎做 before/after 对比）

不依赖 LLM 评委的主观判断，而是**用 p16-reader 自己的模拟读者引擎**对修改版本重新评分。

#### 验证流程

对每个修改版本：

1. 用 E4-1 的 `build_evaluation_prompt` + `call_llm`（temperature=0.3, n_calls=3），让**目标画像**重新评价修改版本
2. 对比修改前后目标画像在该维度的评分

#### 验证 1 — 目标维度提升

修改后，目标画像在目标维度上的评分是否提升？

- 方法：修改前的评分（来自 E4-1 raw） vs 修改后的评分（新建评价）
- 标准：**≥ 2/3 的修改版本在目标维度上有显著提升**（均值提高 ≥ 0.5）

#### 验证 2 — 非目标维度不受损

修改后，目标画像在其他维度上的评分是否保持稳定？

- 方法：对比修改前后目标画像在所有 4 个维度上的评分变化
- 标准：非目标维度的评分**下降超过 0.5 的维度数 ≤ 1**（修改不能拆东墙补西墙）

#### 验证 3 — 画像特异性

针对 P1 的修改和针对 P3 的修改，是否对各自的画像产生了特异性效果？

- 方法：
  - 修改版_P3 → 让 P3 评分 → 预期 P3 提升
  - 修改版_P3 → 让 P1 评分 → 预期 P1 不提升或下降
  - 修改版_P1 → 让 P1 评分 → 预期 P1 提升
  - 修改版_P1 → 让 P3 评分 → 预期 P3 不提升或下降
- 共 4 种组合 × n 篇文本
- 标准：**≥ 3/4 的组合符合预期方向**

### 成功标准

| 验证 | 标准 | 方法 |
|:---|:---:|:---|
| 目标维度提升 | ≥ 2/3 修改版本目标维度提升 ≥ 0.5 | E4-1 评价引擎 before/after |
| 非目标维度不受损 | 非目标维度降幅 > 0.5 的维度 ≤ 1 | 同上 |
| 画像特异性 | ≥ 3/4 组合符合预期方向 | 交叉评价（P3 修改→P1 评） |

### 依赖

- 前置：Phase I 全部通过（E4-0/1/2✅）
- 数据：e4-1_raw.json（已有，作为基线评分）
- LLM 调用量：
  - Step 2 生成：6 篇 × 2 目标画像（P1, P3）× 2 prompt 风格 = ~24 次
  - Step 3 验证评价：最多 24 个修改版本 × 2 画像（特异性验证）× 3 call = ~144 次
  - 总计：~170 次 LLM 调用

### 估计耗时

编码 1 天 + 运行 ~2 小时 + 分析 1 天
