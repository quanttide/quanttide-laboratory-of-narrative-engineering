# E4-7 — 基于多画像评价的局部修改建议（Rewrite）

> 对应能力：Rewrite（修复层）

## 动机

p16-reader 能告诉作者"P3 觉得这个场景套路感强"——但作者需要的是具体怎么改。

E4-7 的核心假设是：**给定文本 + 多画像评分分布 + 目标画像，LLM 能定位薄弱点并生成针对该画像的结构性修改，且修改后目标画像的评分会提升。**

## 实验范围

E4-6（Reflect/视角译码）已被砍掉。E4-7 自包含 Reflect+Rewrite 两个步骤：

```
E4-7 Step 1:      找出"哪里不对"（定位薄弱点）
E4-7 Step 2:      生成"怎么改"（修改版本）
E4-7 Step 3:      验证"改好了吗"（E4-1 评价引擎 before/after 对比）
```

## 核心洞察

有效的修改建议需要满足：
1. **可执行**：作者知道改哪个位置、改什么
2. **有理由**：附带"为什么要这么改"（基于读者画像偏好）
3. **最小改动**：只改薄弱点，不重写全篇
4. **选择性**：不是所有文本都需要改

## 实验设计

### 输入

| 来源 | 内容 | 格式 |
|------|------|------|
| `e4-1_raw.json` | 5 画像 × 6 文本 × 3 call 的完整评分 | JSON |
| `assets/fiction/4_成稿/` | 6 篇测试文本的全文 | Markdown |
| `profiles.json` | 5 个画像的行为锚定定义 | JSON |

### Step 1: 薄弱点定位（自包含 Reflect）

对每篇文本 × 目标画像：

1. 计算目标画像在当前文本上的 4 维度评分均值
2. 与同一画像在 Phase I 6 篇文本上的历史均值对比
3. 按 **差距大小**（|当前值 - 历史均值|）降序排列各维度
4. 如果所有维度的差距都 < 0.5（阈值），判定"无需修改"，跳过

**为什么用"与历史均值的差距"而不是"评分最低的维度"？** 不同维度的评分分布不同。P3 的 cliche_level 历史均值是 1.72（他很挑剔），emotional_impact 是 5.39（他容易被打动）。在 cliche 上 4.0 是巨大的偏离（差距 2.28），在 emotional 上 5.0 几乎是正常（差距 0.39）。用绝对值排序会错误地选择 emotional_impact（数值低）而非 cliche_level（偏离大）。

```
P3 历史均值（Phase I 6 篇文本）:
  writing_quality: 5.11
  emotional_impact: 5.39
  character_realism: 5.94
  cliche_level: 1.72

2.1（展会再遇）P3 评分：
  cliche_level: 4.0 ← 差距 = 2.28 → 最大偏离，需修改
  emotional_impact: 5.0 ← 差距 = 0.39 → 正常范围
```

输出：{text_id, target_profile, target_dimension, current_score, historical_mean, gap, relevant_segment}

### Step 2: 生成修改（Rewrite）

输入：
1. 文本片段（薄弱点相关的段落）
2. Step 1 输出的薄弱点分析
3. 目标画像的行为锚定
4. 约束：最小改动、保持场景结构

输出：

```json
{
  "target_profile": "P3",
  "target_dimension": "cliche_level",
  "current_score": 4.0,
  "historical_mean": 1.72,
  "gap": 2.28,
  "original_segment": "原文段落...",
  "suggested_rewrite": "改写后的段落...",
  "what_changed": ["删去了第三句的套路化对白", "加入了林远亭的犹豫"],
  "why_for_reader": "P3 对套路最敏感——原版对话模式是'你说我拒绝你再说我接受'，老书虫一眼识破"
}
```

对每篇文本 × 目标画像，生成 2 个不同 prompt 风格的版本。

### Step 3: 验证

用 E4-1 的 `build_evaluation_prompt` + `call_llm`（temperature=0.3, n_calls=3），让目标画像重新评价修改版本。对比修改前后评分。

#### 先验噪声基线

在验证之前，先统计同一画像 × 同一文本 3 次调用的标准差（call 间 SD）。

```
P3 × 2.1 cliche_level 3 次评分: ? 
如果 SD=0 → 任何提升都算信号
如果 SD=0.6 → 需要 ≥ 1.2 才不算噪声
```

验证 1 和验证 2 的门槛根据实际 SD 调整：
- 提升/下降的判断标准 = **2 × 目标画像 × 目标维度的 call 间 SD**（而非固定 0.5）
- 如果 SD=0，提升 0.1 也算通过；如果 SD=0.6，需要 ≥ 1.2

#### 验证 1 — 目标维度提升

修改后，目标画像在目标维度上的评分是否提升？

- 方法：修改前均值 vs 修改后均值
- 标准：**≥ 2/3 的修改版本的提升 ≥ 2 × 该维度 call 间 SD**

#### 验证 2 — 非目标维度不受损

修改不能拆东墙补西墙。

- 方法：对比修改前后目标画像在其他 3 个维度上的评分变化
- 标准：**最多 1 个维度降幅超过 2 × 该维度 call 间 SD**

#### 验证 3（可选）— 画像特异性

针对 P1 的修改和针对 P3 的修改，是否对各自的画像产生了特异性效果？

- 方法（交叉评价）：
  - 修改版_P3 → P3 评分 → 预期提升
  - 修改版_P3 → P1 评分 → 预期不提高（若 P1 上升，说明修改是无差别的质量提升）
  - 修改版_P1 → P1 评分 → 预期提升
  - 修改版_P1 → P3 评分 → 预期不提高或下降（P1 喜欢的甜宠元素通常降低 P3 评分）

- 标准：**≥ 3/4 组合符合预期方向**（注意"预期不提高"和"预期下降"的区别——前者更宽松，后者更严格）

- **此验证为可选**：如果验证 1 和验证 2 通过但验证 3 未通过，结论是"修改有效但不是画像特异的"——这不是失败，是有趣的发现（说明多画像评价的分化主要由文本属性驱动，而非画像驱动的差异化修复）。

### 成功标准

| 验证 | 标准 | 方法 | 优先级 |
|:---|:---:|:---|:---:|
| 目标维度提升 | ≥ 2/3 版本提升 ≥ 2×SD | E4-1 评价引擎 before/after | 必选 |
| 非目标维度不受损 | 降幅 > 2×SD 的维度 ≤ 1 | 同上 | 必选 |
| 画像特异性 | ≥ 3/4 组合符合预期方向 | 交叉评价 | 可选 |

### 依赖

- 前置：Phase I 全部通过（E4-0/1/2✅）
- 数据：e4-1_raw.json（已有，作为基线评分）；e4-1_summary.json（画像历史均值）
- LLM 调用量：
  - Step 2 生成：6 篇 × 2 目标画像（P1, P3）× 2 prompt 风格 = ~24 次
  - Step 3 验证评价：最多 24 个修改版本 × 1 画像（仅目标画像，非交叉）× 3 call = ~72 次
  - 验证 3（如做）× 24 × 1 × 3 = 额外 ~72 次
  - 总计：~96 次（必选）或 ~168 次（含可选验证）

### 估计耗时

编码 1 天 + 运行 ~2 小时 + 分析 1 天
