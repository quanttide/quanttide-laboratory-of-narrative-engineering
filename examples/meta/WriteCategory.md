# 写作范畴 (WriteCategory)

一个用 Lean 4 定义的**自由范畴**，建模写作过程的四个阶段及其变换关系。

## 对象

| 对象 | 含义 |
|------|------|
| `material` | 素材 |
| `outline` | 提纲 |
| `firstDraft` | 初稿 |
| `finalDraft` | 成稿 |

## 态射

### 发展阶段（按提纲流程）

```
 organize      writeFromOutline      finalize
material ───→ outline ──────────→ firstDraft ───→ finalDraft
```

### 直接路径（跳跃）

```
draftFromMaterial         publishFromMaterial
material ─────────────→ firstDraft       material ──────────────→ finalDraft
                           polishFromOutline
                   outline ───────────────→ finalDraft
```

### 循环迭代

`review`、`reflect`、`rewrite` 是 `firstDraft` 上的三个自态射（endomorphism），构成写作修改循环：

```
firstDraft ──review──→ firstDraft ──reflect──→ firstDraft ──rewrite──→ firstDraft
```

三者可任意复合，形成无限多的修改路径。

## 范畴结构

该范畴是**箭图 (quiver) 生成的自由范畴**：

- **态射 = 路径**：从一条或多条首尾相接的边组成
- **合成 = 路径拼接**
- **单位态射 = 空路径**
- **范畴律**由路径拼接的幺半群性质保证

## 示例路径

| 路径 | 含义 |
|------|------|
| `organize` | 把素材整理成提纲 |
| `writeFromOutline ∘ organize` | 先整理提纲，再按提纲写初稿 |
| `finalize ∘ rewrite ∘ reflect ∘ review` | 审阅 → 反思 → 改写 → 定稿 |
| `review ∘ review` | 两轮审阅 |
| `rewrite ∘ reflect ∘ review` | 一个完整的修改循环 |

## 从偏序集到一般范畴

相比之前的版本（thin category，偏序集），本次升级：

1. **允许多条态射**：同一对对象间可以有多个不同的态射
2. **允许自环**：`review`/`reflect`/`rewrite` 是 `firstDraft` 上的非平凡自态射
3. **无限态射集**：由于循环可无限复合，`firstDraft → firstDraft` 有无穷多条路径

## 叙事工程视角：展开提纲的内部结构 — 纤维范畴模型

`outline` 不是一个原子对象，而是一个**纤维范畴 (fibered category)** 的基座。构造一个**忘却函子**：

```
U : NarrativeStruct → WritingStage
```

将每个叙事结构（beat / scene / act / arc）映到其所属的写作阶段。`outline` 上的纤维是 `U⁻¹(outline)` — 该阶段所有可能的叙事结构及其变换构成的范畴：

```
                    restructure
              outline ──────────→ outline
              ⇑        ⇑
          narr    beat    reframe
           ⇑        ⇑
      素材      scene ──→ outline
                   ⇑
                  act
                   ⇑
                  arc
```

关键态射的范畴论解释：

- **素材 → 提纲** = **自由函子 (free functor)**：`F : Material → NarrativeStruct`，将非叙事素材生成初始叙事结构，余代数 (coalgebra) 视角下的"叙事生成"
- **提纲 → 初稿** = **解释函子 (interpretation functor)**：`I : NarrativeStruct → Text`，将叙事结构展演为线性文本，代数 (algebra) 视角下的"文本生成"
- **提纲上的自态射** = **叙事变形 (narrative transformation)** 构成的幺半群 `End(outline)`，包括 `restructure`（同构调整）、`reframe`（换范畴，本质上是自然变换）、`rebeat`（节拍精化，态射的因子分解）
- **内部对象**（beat / scene / act / arc）构成一个**偏序集**（按粒度从细到粗），嵌入到 `NarrativeStruct` 中。每个对象又可递归为其上方的纤维，形成**阶层纤维化 (iterated fibration)**：

```
WritingStage                  WritingProcess
     │                              │
     │ Grothendieck                 │ 每个对象纤维
     │ construction                 │ 展开为子范畴
     ▼                              ▼
NarrativeStruct ─── U ───→ WritingStage
     │                              │
     │ iterated                     │ 递归
     │ fibration                    │ 展开
     ▼                              ▼
  Beat/Scene/...              finer-grained
```

这一结构正是 **Grothendieck 构造** 的实例：对忘却函子 `U` 取 Grothendieck 构造，得到**总范畴 (total category)**，其对象为所有写作阶段 × 叙事粒度的配对，态射保持纤维间的变换可相容。

## 术语科普

以上用到了一系列范畴论概念，这里用写作的直觉逐一拆解：

### 函子 (Functor)

> 从一个范畴到另一个范畴的映射，保持对象和态射的结构。

**写作直觉**：`素材 → 提纲` 不是直接的态射，而是一个函子——它将"一堆笔记"（素材范畴的对象）映射为"一个叙事结构"（提纲范畴的对象），同时保持笔记间的关系映射为叙事结构间的关系。类似"格式转换器"，不仅转格式，还保持内容间的引用关系不变。

### 自由函子 (Free Functor)

> 给定一个底层集合，自由地生成满足某种结构的最小对象。是"忘却函子"的左伴随。

**写作直觉**：从一堆零散的素材"自由生成"一个提纲。比如你收集了 10 篇文章的摘录，自由函子自动将它们归类、排序，生成一个粗略的提纲——**不丢失任何信息**，但**新增了结构**（章节、顺序）。这是伴随关系的左半边：自由生成。

### 代数 (Algebra) 与余代数 (Coalgebra)

> **代数** = 从结构到具体值的解释（折叠/归约）。结构 → 值。  
> **余代数** = 从具体值到结构的生成（展开）。值 → 结构。

**写作直觉**：
- **提纲 → 初稿** 是**代数**：给定一个叙事蓝图，你"折叠"它成为一篇具体的线性文本。每一条提纲条目被翻译成一个段落。这是从抽象到具体的归约。
- **素材 → 提纲** 是**余代数**：给定一堆原始素材，你"展开"它成为有结构的蓝图。每一条素材被生成为一个提纲节点。这是从具体到抽象的生成。

代数与余代数恰好构成一个循环：**展开素材为结构 → 折叠结构为文本**，正是"写作 = 从无序到有序再到成品"的形式化。

### 纤维范畴 (Fibered Category)

> 一个函子 `E → B`，其中 `B` 中态射可以"提升"到 `E` 中，类似光纤在不同层级间映射。

**写作直觉**：把 `WritingStage` 看作**楼层**（素材在一楼，提纲在二楼，初稿在三楼，成稿在四楼）。每个楼层里又可以细分成许多**房间**（beat / scene / act / arc）。纤维范畴就是说：你可以从二楼（提纲）的某个房间出发，通过"提升"操作映射到三楼（初稿）的对应房间——**楼层间的路径和房间内的路径一致**。

### Grothendieck 构造

> 给定一个从范畴到范畴的函子 `F : C → Cat`，可以构造一个总范畴，其对象是 `(c, x)` 其中 `x ∈ F(c)`。

**写作直觉**：这是"楼层 × 房间号"的拼合。写作阶段（楼层）和叙事粒度（房间号）两两配对，形成一个**二维坐标系**。`(outline, beat)` 表示"提纲层的节拍"，`(firstDraft, scene)` 表示"初稿层的场景"。态射则允许你在二维空间中任意移动：可以**跨层**（outline → firstDraft）也可以**层内**（beat → scene）。Grothendieck 构造确保这两种移动可以**交换**：先跨层再细化 = 先细化再跨层。

### 自然变换 (Natural Transformation)

> 两个函子之间的态射，保持源范畴中态射的"形状"。

**写作直觉**：假设你有两种不同的写作方法论，方法 A 和方法 B。方法 A 中的 `素材 → 提纲` 和方法 B 中的 `素材 → 提纲` 可能是不同的函子。**自然变换**告诉你：用方法 A 从素材到提纲再到初稿，与用方法 B 从素材到提纲再到初稿，虽然中间步骤不同，但最终结果之间存在一条"翻译通道"——你可以在保持过程结构的同时，将一个方法的结果转换为另一个方法的结果。

### 伴随 (Adjoint)

> 一对相反方向的函子 `F ⊣ G`，满足某种"最优"关系。最核心的范畴论概念之一。

**写作直觉**：`素材` 和 `提纲` 的关系是伴随的——
- **自由生成**（左伴随）`F`：素材 → 提纲。在"不丢失信息"的前提下，自由生成结构。这是最"宽松"的提纲。
- **遗忘**（右伴随）`U`：提纲 → 素材。扔掉结构，只保留原始内容。

伴随性质保证了："先遗忘再自由生成" ≥ 原始结构（丢失结构后重建不会更精细），"先自由生成再遗忘" = 原始素材（结构不影响原始内容）。这恰好是写作中"素材 ↔ 提纲"来回切换时信息守恒的形式化表达。

### 幺半群 (Monoid)

> 一个集合带上一个结合二元运算和单位元。范畴中单个对象上的所有自态射构成一个幺半群。

**写作直觉**：`firstDraft` 上的自态射集合 `{review, reflect, rewrite}` 以及它们的任意复合（`review ∘ rewrite`、`rewrite ∘ reflect ∘ review` 等）构成一个幺半群。单位元是"啥也不做"（恒等态射）。这精确建模了"在初稿阶段你可以做任意多次任意顺序的修改操作"——**版本历史就是这个幺半群中的一个元素**。

## 对写作云架构的启发

### 1. 状态与操作解耦

```
对象 = 文档状态（素材/提纲/初稿/成稿）
态射 = 原子操作（整理/写作/审阅/改写）
```

每个态射对应一个独立微服务或云函数，状态是持久化数据快照。系统通过操作驱动状态流转。

### 2. 组合性 = 工作流编排

复杂流程（如"初审 → 反馈 → 修改 → 再审 → 定稿"）是原子态射的复合。写作云可提供：

- 有向无环图和循环图的自由组合
- 用户自定义管道：`organize >> write >> repeat(review >> reflect >> rewrite, n) >> finalize`

范畴的**复合运算**天然对应云平台的**工作流引擎**：服务的串联与并行由范畴的 composition 和 universal property 统一描述。

### 3. 无限版本空间由自态射给出

`review` / `reflect` / `rewrite` 是 `firstDraft` 上的自态射。在系统中建模为：

```
每次 rewrite 产生一个新版本（仍属 firstDraft 阶段）
版本历史 = 自态射序列：rewrite₃ ∘ rewrite₂ ∘ rewrite₁
```

无需为"初稿 v2"创建新对象，既保持有限的状态机，又支持无限迭代。

### 4. 从线性审批流到循环迭代流

| 偏序模型（传统 CMS） | 一般范畴模型（写作云） |
|----------------------|----------------------|
| 线性审批流程 | 循环迭代流程 |
| 阶段间至多一个操作 | 同一阶段上多个操作 |
| 不支持回滚 | 任意修改循环可组合 |
| 态射即死板路径 | 态射即灵活管道 |

写作云若只支持线性流程（类似传统文档审批系统），适用范围有限。支持循环迭代 + 多路径跳跃，才能覆盖真实写作场景（自由写作 → 反复修改 → 非线性跳跃）。

### 5. 可形式化验证的工作流

Lean 的类型系统可对工作流进行形式化验证：

```lean
-- 示例：证明一个流程总能到达成稿
theorem canAlwaysFinalize (f : Path material firstDraft) :
  ∃ (p : Path material finalDraft), True := ...
```

写作云可利用范畴模型进行：流程可达性验证、资源依赖分析、冲突检测。这为低代码/无代码写作工作流设计器提供了形式语义基础。
