# p17 — 写作契约 vs 读者回响：日记触发器

不做一个"系统替作者反思"的工具，也不做一个"系统输出材料等作者反思"的工具——做一个**帮作者写日记的工具**。

## 动机

journal 里有一段话：

> 我产生了一个新的困惑，就是为什么这种蓄谋式的重逢它会更适合热闹的环境搭配呢？就是它为什么自然而然地导向了这种场景的需求？

这是作者在**自己写日记时**自然产生的反思。它不是任何系统输出触发的——是作者在梳理自己的创作过程时自己想出来的。

p16 说"P3 觉得套路感强"。作者拿到这个信息后，他的自然反应不是"好，我去改"，而是坐下来写一段日记，把自己的困惑理清楚。

p17 的工作不是替作者完成这个过程，也不是把材料并排然后等作者自己想——**p17 的工作是让作者更容易写这段日记**。

```
作者在某个点上卡住 → 跑 p17 → 拿到日记种子 → 粘贴进 journal → 继续写
```

## 实际执行

### 使用方式

不设计成实验管线。p17 是一个 CLI 工具，作者在写作时遇到卡点就运行一次：

```sh
uv run python -m p17 "docs/journal/fiction/2026-06-07.md" \
  --point "咖啡厅重逢 他要了一杯焦糖玛奇朵"
```

输入：作者正在写的 journal 文件路径 + 一个文本位置描述。

输出：一段 markdown，作者可以**直接粘贴进 journal 文件里**，然后继续写。

### Step 0：读取日记上下文

读取作者当前的 journal 文件，从中提取：

- 作者最近在思考什么（"蓄谋式重逢""拉开距离""人设差异化"）
- 作者自己已经意识到的问题（"写得太熟了""AI 没有读出高智高人设的潜台词"）
- 作者正在尝试的方向（"把社会观察写到女主身上"）

这部分不需要 LLM。用最后 N 行 + 标题匹配即可。

### Step 1：契约标注

同之前——标注该文本点触及了 style/motif/story 的哪些条款。但输出时**不列出所有触及条款**，只选出与作者当前日记上下文最相关的那条。

```
日记上下文：作者在纠结"写得太熟了"
→ 契约中与"社交距离"相关的 boundary 优先展示
→ 与"语言风格"相关的 boundary 次优先
→ 其他不展示
```

### Step 2：读者回响

同之前——映射 p16 评价数据。但同样做上下文筛选：

```
日记上下文：作者困惑"蓄谋式重逢为什么适配热闹环境"
→ 展示 P3 在"套路感"维度上的评分（因为蓄谋 vs 巧合的讨论落在套路感上）
→ 不展示其他维度
```

### Step 3：生成日记种子

这是核心输出。它不是材料并排，也不是分析报告——它是一段**可以直接粘贴进 `docs/journal/fiction/` 的 markdown**。

输出格式：

```markdown
--------
## 来自 p17：焦糖玛奇朵

刚在这里卡住了。看了一下契约和读者数据——

### 你之前的日记（2026-06-06）

你提到在"拉开初始距离"这件事上反复改了好几次版本。从安静的咖啡店到热闹的咖啡店，从巧合到蓄谋。

### 契约在这个点上的提示

style.yaml "叙事母题与核心矛盾" boundary 说：
> 暗恋不能通过"泄露他知道她的信息"来推进关系

当前点："他要了一杯焦糖玛奇朵"——暗示他知道她爱喝什么。

### 读者在这个点上的回响

P3（资深老书虫）cliche_level = 1.72（最低）
P1（甜宠少女）cliche_level = 3.11
差距 1.39

### 一个值得注意的事

你日记里说的"写得太熟了"——契约的这条 boundary 和 P3 的低分
都在同一个方向上：这段的社交距离比你想写的要近。

不是你写得不好，是这个点的默认情感温度比你预期的甜。

--------
```

**格式的三条规则：**

1. **开头可定位**——"来自 p17：XXX"让作者知道这是系统生成的，不是自己写的。粘贴进 journal 后一目了然。
2. **引用作者的话**——"你之前的日记（2026-06-06）"让系统输出和作者已有的思考连接起来，而不是凭空生成。
3. **最后一句是"值得注意的事"**——不是结论，不是建议，是一个提示。作者看到后可以接一句"嗯，确实"或"不，其实不是"——然后继续写下去。

### Step 4：作者在日记里继续写

没有反馈回路。作者把上面的种子粘贴进 journal 后，会继续写：

```markdown
嗯，但我觉得不完全是。我选焦糖玛奇朵不是为了让
他知道她爱喝什么——我只是觉得这个单品名好写。
```

**这才是反馈。** 它以日记条目的形式存在于 journal 中，不是以 A/B/C 选项的形式存在。

## 验证

### 验证 1：引用准确

同之前。契约引用和 p16 数据引用必须 100% 准确。

### 验证 2：上下文相关

系统输出的"提示"必须与作者当前的日记上下文相关。不输出作者已经知道的东西。

| 指标 | 标准 |
|:---|:---:|
| 引用的日记上下文是最近的 | 从当前 journal 文件的最近 30 行内提取 |
| 选择的契约条款与日记上下文匹配 | 人工判断：相关 |
| 选择的读者数据与日记上下文匹配 | 人工判断：相关 |

### 验证 3：可粘贴

| 指标 | 标准 |
|:---|:---:|
| 格式是纯 markdown | 100%——无特殊字符、HTML、代码块 |
| 开头有"来自 p17：XXX"标记 | 100%——作者一眼就知道是系统生成的 |
| 以"值得注意的事"结尾 | 100%——不附带结论 |

### 验证 4：有用性

不通过评分测量。通过 journal 本身测量：

| 指标 | 标准 |
|:---|:---:|
| 种子被粘贴进 journal | 记录：作者是否真的把输出贴进去了 |
| 种子后面有作者接的句子 | 记录：种子后面是否有至少一行作者自己的话 |
| 作者接的话里包含新的思考 | 人工判断：作者是否写出了种子之外的新内容 |

**验证 4 是最关键的。** 如果作者贴了种子但没在它后面写任何东西——材料没触发反思。如果作者贴了种子并在后面写了新想法——p17 生效了。种子本身的质量不重要，种子之后的 journal 内容才重要。

### 成功标准

| 验证 | 指标 | 标准 |
|:---|:---:|:---:|
| 引用准确 | 契约 + p16 数据 | 100% |
| 上下文相关 | 相关 | 100% |
| 可粘贴 | 纯 markdown | 100% |
| 触发日记 | 种子后有作者新写的内容 | ≥ 2/4 文本点 |

## 不依赖

纯 Python 脚本 + LLM API。依赖：
- `style.yaml` / `motif.yaml` / `story.yaml`
- p16 的 `e4-1_raw.json`
- 场景原文
- 作者的 journal 文件

## 与 journal 的关系

p17 的整个流程围着 journal 转：

```
作者写日记 → 遇到卡点 → 跑 p17（传入当前 journal 文件）
                        ↓
                  p17 读取日记上下文 → 契约标注 → 读者回响
                        ↓
                  生成日记种子（markdown 块）
                        ↓
作者把种子粘贴进 journal → 在种子后面继续写 → 日记变长了
```

验证 4 的指标直接在 journal 里可读——不需要额外的实验数据采集，不需要创作者评分表。打开 `docs/journal/fiction/` 就能看到哪些种子触发了反思、哪些没有。

## 结果预期

- **T2（焦糖玛奇朵）最有可能触发日记**——因为它正好撞在作者自己也在困惑的点上（"社交距离"）。系统提示和作者的日记上下文天然对齐。
- **T4（假装看菜单）最不可能触发日记**——它是"对齐点"，契约和读者感受一致。作者看到种子后的反应更可能是"嗯，知道了"然后继续写之前就在写的东西。这是好事——不是每个点都需要反思。
- **"引用的日记上下文"可能被作者认为断章取义**——如果系统从日记中提取了一句话但作者认为"我当时不是这个意思"，种子的开头就错了。这不是假阳性，而是提醒作者"你之前写的那句话在别人看来是这个意思"——这本身是一个有价值的信号。
- **关键风险**：作者不完全通过 CLI 使用 p17。如果种子生成的体验比手动复制粘贴麻烦，作者不会用。CLI 命令需要设计得非常轻：`uv run python -m p17 "path" --point "位置"`，输出直接到 stdout 可复制。
- **失败条件**：种子被粘贴后，作者在后面只写了"嗯"或"好的"——说明种子没有提供新的信息量，作者不需要停下来想。
