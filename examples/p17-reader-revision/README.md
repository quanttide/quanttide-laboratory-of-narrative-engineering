# p17 — 写作契约 vs 读者回响：单点反思

验证给定一个具体文本位置 + 作者的写作契约（style.yaml / motif.yaml / story.yaml）+ 多读者的评价数据，LLM 能否在该点上生成对作者有启发的反思——揭示"作者想做什么"与"读者实际感受到什么"之间的对齐或偏离。

## 动机

p16 建立了模拟读者系统。它能告诉作者"P3 在这个场景上觉得套路感强"。

但作者拿到这个信息后，他的自然问题是：**"为什么？我哪里让 P3 觉得套路了？我本来想做什么？"**

前两个问题可以靠 p16 + p14 回答（"焦糖玛奇朵那段，行为链缺了内隐动机"）。第三个问题才是核心——**"我本来想做什么"** 只能靠作者自己的写作契约回答。

style.yaml 中"叙事母题与核心矛盾"维度有一条 boundary：
> "暗恋不能通过'泄露他知道她的信息'来推进关系"

如果作者在这条 boundary 上写了焦糖玛奇朵这个细节，读者 P3 判定为套路——那不是 P3 的问题，也不是作者"写得不好"，而是：**作者想用这个细节表达"他记得她"（暗恋的深情），但 P3 读到的却是"他知道她"（认知泄露的套路）。**

p17 的工作不是替作者想清楚这个差距意味着什么——而是把契约和读者回响并排放在作者面前，让作者自己完成那个"哦原来是这样"的瞬间。

```
                       写作契约 (style / motif / story)
                              ↑
   文本中的一个点 → 并排呈现 → 作者在自己的脑子里完成反思
                              ↓
                    读者感受 (p16 评价数据)
```

```
                       写作契约 (style / motif / story)
                              ↑
   文本中的一个点 → LLM 反思 → 反思：对齐/偏离/张力
                              ↓
                    读者感受 (p16 评价数据)
```

## 核心假设

1. **写作契约可操作**：style.yaml 的 boundaries、motif.yaml 的母题定义、story.yaml 的 tensions，可以作为一个具体文本位置的"预期行为"参照——在该位置上，契约期望作者做什么、不做什么
2. **读者回应可映射**：p16 的读者评价可以映射到同一个具体位置——在该位置上，不同读者实际感受到了什么
3. **并排即信息**：当契约期望和读者感受存在差距时，把两者并排呈现本身就构成了作者反思的条件——不需要 AI 替作者总结"这意味着什么"
4. **反思是作者自己的事**：p17 的输出是材料不是结论。作者看到材料后在自己脑子里完成的那个"哦"才是反思

## 数据来源

### 文本点

从 p16 已评价的 6 篇文本中，选取 4 个具体的文本点。选点标准：p16 评价数据显示**不同画像分歧大**且**触及 style.yaml 的边界**。

| # | 文本点 | 所在场景 | 选点理由 |
|---|-------|---------|---------|
| T1 | "他掏出刚在超市买的毛巾，不知不觉伸出去擦了擦她湿漉漉的头发。回过神来，赶紧把毛巾递给她。" | 4.1 咖啡厅重逢 | p14 标记为"有意断裂"，style 中"情感表达"维度 boundary（身体比脑子快 → 缩回）的最佳范例。读者分歧：这是"深情"还是"越界"？ |
| T2 | "他要了一杯焦糖玛奇朵，帮她拉行李到桌旁。" | 4.1 咖啡厅重逢 | style 中"叙事母题与核心矛盾"的 boundary（暗恋不能通过认知泄露推进）。p16 数据：P3 在此处的套路感评分高。 |
| T3 | "她说'我更心疼在他们的年纪在风暴中孤独无助的你。如果我当时走过去该多好。'" | 10.3 阳台看星星 | 触及"情感表达"维度的边界——直接台词说心疼 vs "情感藏在动作里"的契约。读者分歧：感动 vs 说教。 |
| T4 | "他掏出手机假装看菜单，实际上是在掩饰不知道该怎么面对她的紧张。" | 2.3 傍晚小龙虾 | style 中"对话模式"维度的 boundary（情感转折点在动作和沉默里，不在台词里）。该点用动作暴露内心，与契约高度对齐。 |

**选址原则**：2 个对齐点（契约与读者感受一致）+ 2 个偏离点（契约与读者感受不一致），用来测试反思在不同条件下的表现。

### 输入

| 输入 | 来源 | 在 p17 中的角色 |
|------|------|---------------|
| 文本点原文 | `assets/fiction/4_成稿/` 中的 .md 文件 | 被反思的锚点 |
| 该点前后 2-3 句上下文 | 同上 | 帮助理解语境，避免脱离上下文误读 |
| `style.yaml` | `assets/fiction/` | 作者的审美契约——该点触发了哪些维度的 boundaries |
| `motif.yaml` | `assets/fiction/` | 作者的母题契约——该点涉及/偏离了哪些母题 |
| `story.yaml` | `assets/fiction/` | 作者的角色契约——该点的人物行为与角色定义一致吗 |
| p16 各画像在该场景该维度的评分 | `data/output/e4-1_raw.json` | 读者回响——不同读者在该点所在场景的感受 |
| 无画像基线（P0）的评分 | 同上 | 基准线——没有读者立场时的"中性"评价 |

## 实验步骤

### Step 1：契约标注

对每个文本点，标注它触及了写作契约的哪些条款。

**从 style.yaml 中提取：**

```
文本点：他要了一杯焦糖玛奇朵，帮她拉行李到桌旁。

触及维度：叙事母题与核心矛盾
触及 boundary："暗恋不能通过'泄露他知道她的信息'来推进关系"
触及性质：违反——直接点单的行为暗示"他知道她爱喝什么"，这是认知泄露而非暗恋暴露

触及维度：情感表达
触及 boundary："压抑期的情感暴露只能通过无意识的动作"
触及性质：边缘——行为本身是日常动作，不涉及暴露，但"帮她拉行李"在陌生人语境下偏亲近
```

**从 motif.yaml 中提取：**

```
触及母题：手势（weight 9）
使用方式：拉行李是手势动作，但强度弱——不是"情感微小爆破"，而是日常帮助。与该母题的核心定义（"最高级的情感表达在手里"）有偏差
```

**从 story.yaml 中提取：**

```
触及角色：林远亭
角色定义：沉默寡言、深情但不苦情
行为一致：拉行李是日常行为，不涉及情感表达，不违反角色设定

触及该场景预定义的 tensions：
  - "社交陌生人 vs 心理熟人"——点单行为偏向"心理熟人"（他知道她爱喝什么），弱化了陌生人的张力
```

输出：`契约标注`——{文本点, 触发的 style 维度+boundary, 触及的母题, 相关的角色定义+tension}

### Step 2：读者回响映射

对每个文本点，映射读者在该位置所在场景的相关评分。

```
文本点：他要了一杯焦糖玛奇朵（位于 4.1 咖啡厅重逢）

该场景各画像评分：
              writing  emotional  character  cliche
P1 甜宠少女    4.33     5.00       4.17      3.11
P2 职场丽人    4.72     4.56       4.44      3.17
P3 资深老书虫  5.11     5.39       5.94      1.72
P4 消遣型      4.39     5.06       4.94      3.00
P5 成熟婚恋    4.72     4.72       4.78      3.22

该点的关键读者信号：
  - cliche_level 方差最大（P3 仅 1.72 vs P1 的 3.11）→ 不同读者对套路感的感知差异大
  - P3 在 character_realism 上评分最高（5.94）→ 即使觉得套路，角色真实感不受影响
  - 无基线 P0 未参与该场景评价 → 无"中性"基准，所有评价都有画像 bias
```

输出：`读者回响`——{文本点, 所在场景的各画像评分, 关键信号（方差最大的维度/分歧最大的画像对/异常值）}

### Step 3：材料并排

这是核心步骤。LLM 不生成反思——它只做一件事：**把契约立场和读者回响并排放在一起，停在那里。**

输出格式只有两个部分，没有第三部分：

```
────────────────────────────────────────────
位置：4.1 咖啡厅重逢 "他要了一杯焦糖玛奇朵"
────────────────────────────────────────────

## 契约立场

style.yaml "叙事母题与核心矛盾" boundary 说：
  "暗恋不能通过'泄露他知道她的信息'来推进关系。"

该点判断：他直接点了焦糖玛奇朵，没有问她的口味——
这个行为暗示"他知道她爱喝什么"，属于认知泄露，不是暗恋暴露。

## 读者回响

在该场景上：
  P3（资深老书虫）cliche_level = 1.72（最低）
  P1（甜宠少女）cliche_level = 3.11
  差距 1.39，是全部分歧中最大的画像对

P3 读到的："他知道她爱喝什么"——甜宠文套路，男主万能体贴
P1 读到的："他记得她爱喝什么"——被记住的甜蜜感

────────────────────────────────────────────
这里没有结论。这是作者需要自己看的东西。
────────────────────────────────────────────
```

两个模块不附带任何分析、总结、建议。不写"这意味着……"，不写"建议……"，不写"需要注意……"。

输出止于材料本身。反思发生在作者的脑子里，不在输出的文本里。

### Step 4：作者反馈

作者看到材料后，系统的输出没有结束。作者需要能告诉系统：这个并排有没有用、准不准。

反馈格式（三选一 + 可选文本）：

```
对这个点的反馈：

[ A ] 契约解读准确，我看到差距了（最有价值的反馈）
[ B ] 契约解读准确，但这不是差距——这里没有张力（系统灵敏度过高）
[ C ] 契约解读不准确——我的原意不是这样（系统理解错了）

补充说明（可选）：____________
```

反馈不用于"修正"系统的输出——输出已经被作者看到了。反馈用于：

| 反馈 | 系统的处理 |
|------|-----------|
| A | 记录该选点模式为"有效"，后续类似选点优先使用 |
| B | 降低类似文本点的"差距检测灵敏度"（当前阈值为 1.0 方差，下次调高至 1.2） |
| C | 将契约解读记录为"需要人工仲裁"——如果同一个 boundary 被多次误读，说明该 boundary 的表述需要作者重新写

## 验证

### 验证 1：契约引用准确

| 指标 | 标准 | 方法 |
|:---|:---:|:---|
| 触发的 boundary 识别准确 | 100% | 与人工标注对比——每个文本点实际触及哪些 style boundaries |
| 母题关联不编造 | ≥ 80% | 输出中提及的母题必须在 motif.yaml 中有定义 |
| 角色引用不断章取义 | 100% | 引用 story.yaml 中的角色描述时不歪曲原意 |

### 验证 2：读者数据不误读

| 指标 | 标准 | 方法 |
|:---|:---:|:---|
| p16 数据引用准确 | 100% | 输出中引用的评分数据与实际 e4-1_raw.json 一致 |
| 分歧画像对识别正确 | 100% | LLM 标注的"分歧最大的画像对"与数据计算一致 |

### 验证 3：不越界

**这是最重要的验证。** 系统不能替作者完成反思。

| 指标 | 标准 | 方法 |
|:---|:---:|:---|
| 无分析句 | 输出中不包含"这意味着……""说明……""因此……"等总结句 | 100% 的文本点通过 |
| 无建议句 | 输出中不包含"建议……""可以……""需要注意……"等指导句 | 100% 的文本点通过 |
| 无结论句 | 输出不以判断句结尾（"这是张力的""这是偏离的"等） | 100% 的文本点通过 |

### 验证 4：可用性（创作者评估）

| 指标 | 标准 |
|:---|:---:|
| "看完之后我自己想到了之前没注意到的东西" | ≥ 2/4 文本点 |
| "契约解读与我本意一致"（1-5） | ≥ 4 |
| "系统没有替我想，这很好"（1-5） | ≥ 4 |
| "反馈机制有用——我能告诉系统它准不准"（1-5） | ≥ 3.5 |
| "整体有用性"（1-5） | ≥ 3.5 |

### 成功标准

| 验证 | 指标 | 标准 |
|:---|:---:|:---:|
| boundary 识别 | 与人工一致 | 100% |
| 母题不编造 | 提及的母题在 motif.yaml 中定义 | ≥ 80% |
| p16 数据引用 | 与实际一致 | 100% |

## 不依赖

纯 Python 脚本 + LLM API。依赖：
- `style.yaml`（审美维度 + boundaries）
- `motif.yaml`（母题定义）
- `story.yaml`（角色描述 + 场景 tensions）
- p16 的 `e4-1_raw.json`（读者评价数据）
- 场景原文（`assets/fiction/4_成稿/` 中的 .md 文件）
- p16 的 `profiles.json`（画像定义，用于解释"P3 是谁"）

## 与 p16 的关系

p17 不是 p16 的替代或扩展——它们是作者工作台上的两个独立工具：

| 维度 | p16 | p17 |
|------|-----|-----|
| 问题 | 不同类型的读者怎么看这个文本？ | 在这个点上，我的契约和读者的感受之间有什么差距？ |
| 输出 | 评价分布（各画像评分 + 统计检验） | 材料并排（契约立场 + 读者回响，无结论） |
| 使用者 | 作者想看读者反馈时 | 作者在某个具体点上卡住时 |
| 验证 | η² / ICC / 分类准确率 | 引用准确率 + 不越界检查 + 有用性评分 |
| 价值 | "P3 觉得套路感强" | 作者自己看完说"哦，原来契约和读者在这个点上分叉了" |

## 结果预期

- **边界识别（验证 1）预期全通过**——style.yaml 的 boundaries 是显式规则，LLM 逐条比对即可，不是推理任务。
- **T1（毛巾缩回）和 T4（假装看菜单）的材料最无趣也最有用**——这两个点契约与读者感受一致，材料呈现出的信息是"这里没有差距，你做得对"。虽然平淡，但"确认自己正走在契约里"对作者有价值。
- **T2（焦糖玛奇朵）的材料最有反思空间**——契约说这是认知泄露，P3 读出了套路，但作者想表达深情。三者之间有一个真正的差距要作者自己去想。
- **T3（直接台词说心疼）对"不越界"的考验最大**——LLM 在这里最容易忍不住写"这说明……"或"建议……"。如果验证 3（不越界检查）在这里失败了，说明 prompt 还需要收紧。
- **反馈回收率是关键指标**——如果作者懒得给反馈（回收率 < 75%），说明反馈机制太麻烦或者材料本身不值得回应。需要简化反馈格式或检查选点质量。
- **C 类反馈（契约解读不准确）是最有价值的问题信号**——它不是系统的失败，而是契约本身的模糊点：如果同一段 boundary 被多次误读，说明该 boundary 的表述有歧义，需要作者重新澄清。这是"框架可演化"假设的实际体现。
- **关键风险**：作者不在实验回路中。验证 4（可用性）依赖原型用户的审美判断力。如果原型用户不是真正的作者（对 style 没有归属感），可用性评分和反馈质量可能偏低。
- **失败条件**：如果作者看完材料后的反应是"所以呢？"——说明契约和读者回响的并排本身没有产生信息量。可能是选点不对（选了没有差距的点），也可能是并排方式不够直观。
