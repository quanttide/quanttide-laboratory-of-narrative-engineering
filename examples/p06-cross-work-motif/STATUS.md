### 三、⚠️ 代码缺陷：两个重要的信息泄露

#### 缺陷 1：母题链重构（步骤 3）泄露了 GT 母题标签

**代码**（`run.py` 第 187 行）：
```python
f"[{f['motif']}] {f['series']}-{f['scene']}: {f['description']}"
```

**问题**：给 LLM 的每个场景片段前面附带了 `[手势先于语言]`、`[日常的缝隙]` 等 GT 母题名称。

**READM 设计要求**：
> "不使用 gallery 的母题标签，将 14 个场景片段直接提交给 LLM"

**影响**：LLM 不需要任何聚类能力——它直接看到每个片段的标签，按标签分组即可。5/6 完全匹配的结果没有信息量。

**修复**：改为 `{f['series']}: {f['scene']}: {f['description']}`，去掉 `[f['motif']]`。

#### 缺陷 2：盲品配对（步骤 4）泄露了系列信息

**代码**（`run.py` 第 233 行）：
```python
f"[{i+1}] 系列: {'都市' if f['series']=='urban' else '校园'}\n    描述: {f['description']}"
```

且 prompt 明确告诉 LLM：
> "每组 2 个场景应当体现同一叙事母题（一个来自都市言情，一个来自校园言情）"

**READM 设计要求**：
> "打乱顺序，去掉所有标签（母题名、作品名、场景名）"

**影响**：LLM 知道每组一定包含一个都市 + 一个校园，8 个场景 = 4 个母题 → 只需将 4 个校园匹配给 4 个都市。搜索空间从 105 种分组降为 24 种配对。**100% 的准确率不能完全反映 LLM 的跨作品识别能力。**

**修复**：在 prompt 中完全去掉系列标识，且不告诉 LLM 每组必须含一个都市 + 一个校园。

这种**动作结构（action→realization→backtracking/continuation）** 是 LLM 擅长的模式识别类型。**这对产品是好消息**：如果母题识别可以基于结构化动作序列而非纯抽象概念，LLM 的跨作品映射将非常可靠。
